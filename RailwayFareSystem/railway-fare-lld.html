🚆 Railway Fare System — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design the fare engine for a train route. Stations sit in order — A → B → C → D → E — each a known distance apart. A passenger boards at one station and gets off at another; we need the distance between them, then charge a fare based on that distance and the compartment class they chose (2nd class, 1st class, or AC).

ACTORS Who uses it?

  • Passenger — picks board & destination stations + a class.
  • System — finds the distance, computes the fare.

ENTITIES What "things" exist?

  • Station, Route, Train
  • CompartmentClass (2nd / 1st / AC)
  • Fare rule per class

BEHAVIOR What happens? (the flow)

pick from & to → distance = km(to) − km(from) → fare = classRule(distance)

Two clean sub-problems: (1) the distance lookup along the route, and (2) the fare rule, which differs per class → a natural Strategy.

🧩 Design pattern in focus

Strategy pattern — each compartment class has its own fare formula behind a FareStrategy interface (2nd = per-km, AC = base + per-km…), so pricing per class can change or a new class be added without touching the others. Built in requirement NF1. The cumulative-distance route model is the other key idea.

🔗 Relationships (in plain words)

──▷
IS-A: SecondClassFare / FirstClassFare / AcFare is a FareStrategy.
◆──
HAS: a Route has ordered Stations (+ their km); a Train has a Route; FareService maps each class → its FareStrategy.
- -▷
USES: FareService asks the Route for the distance, then the class's FareStrategy for the fare.

🌊 End-to-end flow

fare("B", "D", AC)
  └─ distance = route.distanceBetween("B","D") = 120 − 50 = 70 km
  └─ strategy = fareByClass.get(AC) // pick the AC formula
  └─ strategy.calculate(70) = 50 + 70×3 = ₹260 🎟️
