🎬 Movie Ticket Booking — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design a movie ticket booking system (BookMyShow-style). A user browses shows (a movie on a screen at a time), sees which seats are free, picks some, and pays. The one rule that must never break: two users can never book the same seat — even if they click at the exact same moment.

ACTORS Who uses it?

  • User — browses shows, picks seats, pays.
  • System — tracks availability, books atomically, prices it.

ENTITIES What "things" exist?

  • Movie, Theatre, Screen, Seat
  • Show (movie + screen + time)
  • User, Booking

BEHAVIOR What happens? (the flow)

pick a show → see free seats → select seats → book (atomically!) → pay → confirmed

The distinctive challenge isn't a GoF pattern — it's concurrency: making "check the seat is free" and "reserve it" happen as one un-interruptible step.

🧩 Design focus

Concurrency & seat locking — the key challenge is preventing double-booking when many users act at once. The fix is an atomic check-and-reserve (a lock), built in requirement NF1. (Not a GoF pattern; pricing could optionally use Strategy.)

🔗 Relationships (in plain words)

◆──
HAS: a Screen has Seats; a Show references a Movie + a Screen and owns its own bookedSeats set; a Booking has seats + amount.
1—*
ONE-TO-MANY: one Movie plays in many Shows; one Show produces many Bookings.
🔒
ATOMIC: BookingService calls show.tryReserve() which is synchronized — the check & reserve happen as one step.

🌊 End-to-end flow

book(user, show, [A5])
  └─ validate seats exist on the screen
  └─ show.tryReserve([A5]) // synchronized: atomic check + reserve
      ├─ taken? → return FAILED booking
      └─ free? → reserve, price by seat type → CONFIRMED 🎟️
