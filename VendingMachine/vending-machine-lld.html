🥤 Vending Machine — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design the software for a vending machine. A customer inserts coins, selects a product, and the machine dispenses it — returning change if they overpaid. It must refuse when a product is out of stock or the money isn't enough, and allow a refund. Crucially, the machine responds differently to the same button depending on its current situation (before money, after money, mid-dispense).

ACTORS Who uses it?

  • Customer — inserts coins, selects, collects product + change.
  • Machine — validates, dispenses, returns change/refund.

ENTITIES What "things" exist?

  • Vending Machine, Inventory
  • Product (code, name, price)
  • Coin, Balance, Change

BEHAVIOR What happens? (the flow)

insert coins → select product → (enough money & in stock?) → dispense + change → reset

Notice the "situation matters": selecting before paying, or paying mid-dispense, must behave differently. That hint points straight at the State pattern later.

4 · Final integration

All pieces combined. Here's the complete class diagram, the state machine, the relationships, and the end-to-end flow.

📦 Class diagram

- products (Map)

«interface» VendingMachineState

↑ Idle, HasMoney, Dispense
VendingMachine «context»
- currentState- inventory, balance- selectedProduct
+ insertCoin / selectProduct+ dispense / refund

🔁 The state machine (heart of the design)

IDLE ──insertCoin──▶ HAS MONEY ──selectProduct──▶ DISPENSING ──dispense──▶ IDLE

Also: HAS MONEY ──refund──▶ IDLE. Each state is a class; each arrow is a state transition triggered by a user action.

🔗 Relationships (in plain words)

──▷
IS-A: IdleState / HasMoneyState / DispenseState is a VendingMachineState.
◆──
HAS: VendingMachine owns an Inventory, the 3 state objects, and a pointer to the current one. Inventory holds Products.
- -▷
USES / calls back: each State calls back into VendingMachine to change balance, stock, and switch state.

🌊 End-to-end flow

insertCoin(quarter)
  └─ machine delegates → IdleState.insertCoin() → balance += 25, state = HasMoney
selectProduct("A1")
  └─ HasMoneyState checks stock + price → ok → state = Dispensing
dispense()
  └─ DispenseState → drop product + change → reduce stock → reset → state = Idle 🥤
