🛗 Elevator System — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design the software controlling a building's elevators. People press buttons — either a hall call on a floor (UP/DOWN) or a car call inside (a destination). Each car must move to serve its stops efficiently. When there are several cars, the system must decide which car answers each request.

ACTORS Who uses it?

  • Rider — presses hall/car buttons, boards, gets off.
  • System — assigns a car, moves it, opens doors.

ENTITIES What "things" exist?

  • Elevator (car), Floor
  • Request (hall call / car call)
  • System controller, Scheduling rule

BEHAVIOR What happens? (the flow)

request → controller picks a car → car adds the stop → car moves & opens doors → idle when done

Two decisions to design: which car takes a request (scheduling), and how a car orders its stops as it moves (a mini SCAN).

🔗 Relationships (in plain words)

──▷
IS-A: NearestElevatorStrategy / LeastBusyStrategy is a SchedulingStrategy.
◆──
HAS-MANY: ElevatorSystem owns a List<Elevator>; each Elevator owns its TreeSet of stops.
- -▷
USES: ElevatorSystem uses a SchedulingStrategy to pick a car for each Request.

🌊 End-to-end flow

requestElevator(Request floor=5)
  └─ strategy.selectElevator(req, cars) → nearest car
  └─ car.addStop(5) → stop remembered, direction set
// each clock tick:
system.step()
  └─ every car.step() → move 1 floor toward next stop → doors open on arrival 🛗
