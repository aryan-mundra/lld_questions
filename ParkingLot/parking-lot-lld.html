🅿️ Parking Lot — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design the software for a parking lot. Vehicles enter, are given a place to park, and pay a fee based on how long they stayed when they leave. The lot has multiple floors, spots come in different sizes, and different vehicles need different-sized spots.

ACTORS Who uses it?

  • Driver — brings a vehicle, wants to park & pay.
  • System — assigns spots, issues tickets, charges fees.

ENTITIES What "things" exist?

  • Parking Lot, Floor, Spot
  • Vehicle (Car / Motorcycle / Truck)
  • Ticket, Fee

BEHAVIOR What happens? (the flow)

enter → find a fitting free spot → park & get ticket → (time passes) → exit → free spot → pay fee

If you can narrate this flow, you understand the system. Everything we design will serve this flow.

🔗 Relationships (in plain words)

──▷
IS-A: Car / Motorcycle / Truck is a Vehicle. HourlyFee is a FeeStrategy.
◆──
HAS-MANY (owns): ParkingLot has a List<ParkingFloor>; ParkingFloor has a List<ParkingSpot>.
◇──
REFERS-TO: Ticket points to a Vehicle and a ParkingSpot.
- -▷
USES: ParkingLot uses a FeeStrategy on exit.

🌊 End-to-end flow

park(car)
  └─ lot loops floors → floor.findAvailableSpot(car)
      └─ spot.canFit(car)? → yes → spot.park(car)
  └─ create Ticket(car, spot, now) → return to driver
// ...time passes...
exit(ticketId)
  └─ spot.remove()   +  feeStrategy.calculate(ticket) → amount 💰
