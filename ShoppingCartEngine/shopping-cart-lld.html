🛒 Shopping Cart Engine — Low Level Design

Learn by building it the way an interviewer expects: understand → gather requirements → design one requirement at a time → integrate → dry run.

1 · Understand the problem

Before any code, say the problem out loud in plain English. Build intuition first.

📝 Problem statement

Design the engine behind an online shopping cart. A shopper adds products (with quantities), can change or remove them, and sees a running total. At checkout the price gets adjusted by a stack of charges and discounts — member discount, tax, coupons, gift wrap — applied in a specific order. New offer types get added all the time, so the pricing must be easy to extend.

ACTORS Who uses it?

  • Shopper — adds/updates items, applies coupons, checks out.
  • Engine — tracks items, computes subtotal & final price.

ENTITIES What "things" exist?

  • Product, CartItem, ShoppingCart
  • Bill (subtotal)
  • Charges: tax, coupon, discount, gift wrap

BEHAVIOR What happens? (the flow)

add items → subtotal → apply discount → +tax → −coupon → +gift wrap → final total

The charges stack on top of each other, each transforming the running amount. That "wrap the price in layer after layer" shape points straight at the Decorator pattern.

🧩 Design pattern in focus

Decorator pattern — each charge/discount is a wrapper that takes a Bill and returns a Bill, adjusting the amount. Stack them in any order; add new charge types without touching the cart or existing charges. You'll build it in requirement NF1.

🧅 The Decorator "onion" (why order matters)

GiftWrap +₹30
Coupon −₹50
Tax +8%
Discount −10%
CartBill → Subtotal

Each layer calls the one inside it, then applies its own twist. Read outward: total = giftWrap( coupon( tax( discount( subtotal ) ) ) ). Re-order the wrapping → different total.

🔗 Relationships (in plain words)

──▷
IS-A: CartBill and every decorator is a Bill. Tax/Discount/Coupon/GiftWrap is a BillDecorator.
◇──
HAS-A (wraps): a BillDecorator holds the Bill it decorates — the recursive core of the pattern (IS-A and HAS-A the same type).
◆──
OWNS: ShoppingCart owns its CartItems; CartBill reads the cart's subtotal live.

🌊 End-to-end flow

Bill bill = new CartBill(cart); // ₹460
bill = new DiscountDecorator(bill, 10); // ₹414
bill = new TaxDecorator(bill, 8); // ₹447.12
bill = new CouponDecorator(bill, 50); // ₹397.12
bill = new GiftWrapDecorator(bill, 30); // ₹427.12
bill.getAmount(); // each layer calls the one it wraps 🛒
